## Current-Config Events + FP Verdicts

- **`block_broad_grep` ×1** (`grep -r foo /tmp/ 2>&1 | head -3`): **FP** (desired behavior). A `head`-bounded output should make the block unnecessary. → **FIX:** exempt grep piped to `head` (and other bounded-output forms). In the spirit of the fine-granularity principle.
- **`block_cd_drift` ×2** (`cd .../worktrees/X && … <no cd-back>`): **LEGITIMATE**. Both chains `cd` into a worktree without cd-ing back; the hook blocked correctly. Verified: in one case the agent then switched itself to `WT=…; git -C $WT …` (the desired pattern). No fix.
- **`block_polling_loop` ×2** (`ps -p <pid>; tail <log>` one-shot check): **FP**. See the design conclusion below. → FIX pending decision.

## polling_loop — Design Conclusion

The original failure (a worker died): repeated SEPARATE `ps`+`tail` calls every few seconds = cross-call frequency. A stateless PreToolUse hook only ever sees one call → it CANNOT detect cross-call frequency without a state file (complex + prone to its own FPs; "not really a hook's job"). The hook at the time matches the FORM of a single `ps`+`tail` command → over-blocks one-shot status checks (FP) AND misses the real repetition failure. Doubly wrong.

Options: (a) narrow to in-command loop syntax — kills the one-shot-check FP, but doesn't catch the REAL cross-call failure (repeated separate calls) at all; (b) stateful frequency-aware; (c) disable.
**Decision 2026-05-29 (user): (b) frequency version.** (a) discarded because it doesn't catch the actual failure; (b) catches it AND avoids the one-shot-check FP. Implemented (see implementation below).

## Kill-Protection Follow-Up (closed)

Proposal: block `tmux kill-session -t worker-*` + bare `kill <claude-pid>`, let the rest through.
**State at the time:** `block_dangerous_kill` covers the class `pkill -f` / `pgrep -f|kill` / `kill $(pgrep)` / `ps|grep|kill` (block message explicitly "risk killing worker sessions — use worker-cli kill"). Fired 4× (menubar `pgrep|xargs kill`, dolt `pkill -9 -f`, workflow `ps|grep|kill`). So the most common worker-kill danger IS protected.
**NOT covered:** `tmux kill-session -t worker-*` (no tmux pattern), bare `kill <numeric claude-pid>` (only pipeline/subst forms match). No log evidence that these cases occurred. → **closed for now**; resurfaces in the recurring audit if it becomes real. No tracking task.

**Bonus FP finding:** `block_dangerous_kill` blocked `pkill -9 -f "dolt sql-server"` (a legitimate dolt restart) — over-block. Relevant to the dolt circuit-breaker topic (restart-kill getting blocked) and the general over-reach follow-up. Candidate fix: exempt known service restarts (dolt) from the `pkill -f` block.

**Side root-cause finding (worker spawn × bd cleanup):** while building these fixes, two workers died via SIGKILL at startup. Cause: bd's "orphaned dolt sql-server" cleanup kills processes by **cmdline match** on the string "dolt sql-server"; worker spawn passes the entire prompt as a command-line argument (`claude "$(cat prompt)"`), and the prompt in fix 2 happened to contain exactly that string → bd mistook the worker for an orphaned dolt process and killed it. Workaround: moved the string out of the prompt into a side file. This also explains why the dolt allowlist is safe (no worker can carry the string without being killed by bd first — self-exempting). **Structural follow-up question (its own topic):** spawn should pass prompts via stdin/file instead of the command line, otherwise every worker prompt with a kill-/cleanup-matchable string is vulnerable.
