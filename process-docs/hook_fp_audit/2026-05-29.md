## polling_loop — Design-Schluss

Ursprünglicher Ausfall (Worker starb): wiederholte SEPARATE `ps`+`tail`-Calls alle paar Sekunden = Cross-Call-Frequenz. Ein stateless PreToolUse-Hook sieht immer nur einen Call → kann Cross-Call-Frequenz NICHT erkennen ohne State-Datei (komplex + selbst FP-anfällig; „eig. keine Hook-Sache"). Der heutige Hook matcht die FORM eines einzelnen `ps`+`tail`-Befehls → over-blockt Einmal-Status-Checks (FP) UND verfehlt den echten Wiederholungs-Ausfall. Doppelt falsch.

Optionen: (a) auf In-Command-Loop-Syntax verengen — killt den Einmal-Check-FP, fängt aber den ECHTEN Cross-Call-Ausfall (wiederholte separate Calls) gar nicht; (b) stateful frequenz-aware; (c) deaktivieren.
**Entscheidung 2026-05-29 (User): (b) Frequenz-Version.** (a) verworfen, weil sie den eigentlichen Ausfall nicht fängt; (b) fängt ihn UND vermeidet den Einmal-Check-FP. Umgesetzt (siehe Umsetzung unten).

## 9g6m — Kill-Protection (geschlossen)

Bead-Vorschlag: `tmux kill-session -t worker-*` + bare `kill <claude-pid>` blocken, Rest durchlassen.
**IST:** `block_dangerous_kill` deckt die Klasse `pkill -f` / `pgrep -f|kill` / `kill $(pgrep)` / `ps|grep|kill` ab (Block-Message explizit „risk killing worker sessions — use worker-cli kill"). 4× gefeuert (menubar `pgrep|xargs kill`, dolt `pkill -9 -f`, workflow `ps|grep|kill`). Die häufigste Worker-Kill-Gefahr IST also geschützt.
**NICHT abgedeckt:** `tmux kill-session -t worker-*` (kein tmux-Pattern), bare `kill <numerische claude-pid>` (nur Pipeline-/Subst-Formen matchen). Kein Log-Evidence dass diese Fälle vorkamen. → **9g6m geschlossen**; taucht im wiederkehrenden Audit wieder auf falls real. Kein Bead.

**Bonus-FP-Befund:** `block_dangerous_kill` blockte `pkill -9 -f "dolt sql-server"` (legitimer Dolt-Restart) — Over-Block. Relevant für **7lp2** (Dolt Circuit Breaker: Restart-Kill wird geblockt) + **v2o7** (Over-Reach). Kandidat: bekannte Service-Restarts (dolt) vom `pkill -f`-Block ausnehmen.

**Root-Cause-Befund am Rande (worker spawn × bd cleanup):** Beim Bauen dieser Fixes starben zwei Worker per SIGKILL beim Start. Ursache: `bd`'s „orphaned dolt sql-server"-Cleanup killt Prozesse per **Cmdline-Match** auf den String „dolt sql-server"; der Worker-Spawn übergibt den ganzen Prompt als Kommandozeilen-Argument (`claude "$(cat prompt)"`), und der Prompt enthielt in Fix 2 genau diesen String → bd hielt den Worker für einen verwaisten Dolt-Prozess und schoss ihn ab. Workaround: String aus dem Prompt in eine Seiten-Datei ausgelagert. Das ist zugleich die Begründung, warum die dolt-Allowlist sicher ist (kein Worker kann den String tragen ohne von bd gekillt zu werden — self-exempting). **Strukturelle Folge-Frage (eigenes Thema):** Spawn sollte Prompts via stdin/Datei statt Kommandozeile übergeben, sonst ist jeder Worker-Prompt mit kill-/cleanup-matchbaren Strings anfällig.
