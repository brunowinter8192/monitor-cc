# NSPanel Refactor — Phase A.1 Investigation

## Problem

`src/menubar/menubar.py` uses `NSStatusItem` + `NSMenu`. When any click lands outside the open menu, AppKit's `NSEventTrackingRunLoopMode` cancels event tracking and dismisses the menu — this is NSMenu's by-design behavior. Goal: replace NSMenu with a persistent NSPanel that only closes on explicit Cmd+L toggle.

## Design Questions Answered

### (a) rumps coexistence — can we keep rumps?

**YES.** The NSMenu is attached exactly once in rumps, at `jaredks/rumps rumps/rumps.py:1093`:
```python
self.nsstatusitem.setMenu_(mainmenu._menu)
```
This is the only `setMenu_` call in all of rumps. Setting it to `None` afterward detaches the NSMenu permanently — rumps never re-calls `setMenu_`. The title (`◉`/`●`), blink logic, `@rumps.timer`, and `quit_button` are unaffected.

After the lazy-init tick (when `_nsapp` is available per rumps `run()` line 1350):
```python
self._nsapp.nsstatusitem.setMenu_(None)
btn = self._nsapp.nsstatusitem.button()
btn.setTarget_(_panel_controller)   # PyObjC NSObject subclass
btn.setAction_(b'togglePanel:')
```

The Carbon hotkey (Cmd+L) calls `button().performClick_(None)`. Per AppKit docs: *"If the button has an attached menu, the menu is shown. Otherwise, the button's action is sent."* With `setMenu_(None)`, `performClick_` fires the button's action → `togglePanel_`. No hotkey code changes needed.

**Discarded alternative — drop rumps entirely:** Would require rewriting NSStatusBar management, @timer wiring, and the run loop from scratch via raw PyObjC. Unnecessary because `setMenu_(None)` suffices to decouple NSMenu while keeping all other rumps infrastructure.

### (b) Dismissal mechanic — how does NSPanel stay open?

**Source:** `iSapozhnik/Popover Sources/Popover/PopoverWindow.swift` lines 39–54 (factory method `window(with:)`):
```swift
styleMask: [.nonactivatingPanel]           // NSWindowStyleMaskNonactivatingPanel = 128
level = .statusBar                          // NSStatusWindowLevel ≈ 25
collectionBehavior = [.canJoinAllSpaces, .ignoresCycle]   // 1 | 64
```

`NSWindowStyleMaskNonactivatingPanel` (128) is the key: the panel appears without activating the app or becoming the key window. AppKit has no modal event-tracking loop to cancel, no key-window-resign trigger — so it has no mechanism to auto-dismiss the panel.

`iSapozhnik/Popover Popover.swift` lines 133–139 shows `keepPopoverVisible: Bool` guarding a `globalEventMonitor` dismiss call. For our case we go further: **no global event monitor at all**. Dismissal is only via explicit `_toggle_panel()` code.

**`bikekoala/here-macos Here/StatusBar/StatusBarController.swift` line 187** (comment confirming window level is non-negotiable):
> *"`.statusBar` level is required, not just an aesthetic choice. NSPopover internally treats positioning views in status-bar-level windows as 'menu-bar-attached' — the popover renders flush under the menu bar."*

Their code at line 153 also notes: *"Backup plan: re-implement the popover as a self-managed NSPanel and skip NSPopover entirely"* — confirming the pattern.

**PyObjC integer constants (verified at runtime):**
- `NSWindowStyleMaskNonactivatingPanel` = 128
- `NSStatusWindowLevel` = 25
- `NSWindowCollectionBehaviorCanJoinAllSpaces` = 1
- `NSWindowCollectionBehaviorIgnoresCycle` = 64

**Discarded alternative — global mouse monitor for outside-click dismiss:** `iSapozhnik/Popover` uses `NSEvent.addGlobalMonitorForEvents(.leftMouseDown, .rightMouseDown)` to dismiss on outside clicks. We explicitly do NOT install this because sticky (never-auto-dismiss) is the design goal. Only Cmd+L / bar-icon click close the panel.

### (c) Live-update hook — does NSPanel freeze the runloop?

**No.** `NSEventTrackingRunLoopMode` is exclusive to NSMenu's modal event-tracking loop. `NSPanel.orderFrontRegardless()` is a normal window show — no runloop mode change. `NSDefaultRunLoopMode` stays active. `@rumps.timer(POLL_INTERVAL)` (`_tick`) continues firing at 1.5s while the panel is visible.

**Consequence:** `_MenuDelegate`, `_tracking_timer` (NSTimer at 0.5s, NSRunLoopCommonModes), and `NSRunLoopCommonModes` import are fully eliminated. Three complexity sources (NSMenuDelegate ObjC bridging, NSRunLoopCommonModes scheduling, parallel 0.5s timer) collapse to zero. The `_menu_open` flag (renamed `_panel_open`) still gates `_tick` between full-rebuild and in-place-update paths.

## Decision Conclusion

NSPanel with `NSWindowStyleMaskNonactivatingPanel` + `NSStatusWindowLevel` is the correct direction. Keeps rumps (lower risk, less rewrite), eliminates `_MenuDelegate` complexity, and resolves the runloop freeze. Phase A.2 builds a dev/ probe to verify the three design answers interactively before touching `src/`.
