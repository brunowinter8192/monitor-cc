## Problem

Parallel `tool_use` blocks in one assistant message — multiple Bash, or Bash mixed with Read/Edit/etc. — are not merely wasteful. Failure cascade:

1. Runtime cancels the sibling tool calls when the first errors (`Cancelled: parallel tool call`).
2. Under extended thinking, the cancellation mutates the in-flight `thinking` blocks of the assistant message.
3. Next API request → `400 ... thinking blocks cannot be modified` / `thinking.cache_control: Extra inputs are not permitted`.
4. The broken message stays in conversation history → every subsequent turn 400s → session wedges permanently.

Observed live this session: API 400 on `messages.69.content.3.thinking.cache_control` after a mixed Read + multiple-Bash + Edit batch was cancelled.

## What Was Tried — Both Dead

### Approach 1 — Transcript-read PreToolUse hook (`block_parallel_bash.py`)

Idea: PreToolUse hook reads the CC session JSONL (`transcript_path`) to find the sibling `tool_use` entries of the current message; block (exit 2) if >1.

DEAD. The session JSONL is written only AFTER the full assistant message has round-tripped to the API and back. At PreToolUse dispatch time the current message's `tool_use` lines are not yet in the JSONL — only prior turns are. Reconstruction proof: with both sibling lines artificially inserted, the logic blocks (exit 2); live, they are never present in time. Transcript-as-detection is structurally too late.

### Approach 2 — Cross-invocation timing heuristic

Idea: parallel hooks fire within milliseconds; a per-session state file detects "<200ms since last Bash hook → same batch → block".

REJECTED (user: "kompletter bs"). Race conditions (parallel hooks write the state file simultaneously → file-locking required), and structurally cannot block the FIRST call in a batch (no predecessor to compare against). Not built.

## Fundamental Reason No Hook Works

CC fires ONE PreToolUse hook per tool call. Each invocation receives only its own `tool_input` — never the sibling tool_use blocks of the same assistant message. There is no hook-layer vantage point from which "this message dispatches N parallel tool calls" is visible. The only cross-tool record (the transcript) materializes after dispatch. → No detection point exists before the damage is done.

But even with a correct path the hook could not see its sibling — the approach is dead independent of the path bug.

## Evidence — Why Cancellation Wedges the Session

Anthropic Extended-Thinking docs (Monitor_CC_reference):

- `ExtendedThinking4.md` (Preserving thinking blocks): "During tool use, you must pass thinking blocks back to the API ... include the complete unmodified block. ... the entire sequence of consecutive thinking blocks must match the outputs generated by the model during the original request; you can't rearrange or modify the sequence of these blocks."
- `ExtendedThinking4.md` (caching): interleaved thinking amplifies cache invalidation; thinking blocks can occur between multiple tool calls; thinking blocks must be preserved when continuing tool-use conversations.
- `ExtendedThinking2.md`: round-tripped thinking blocks must be passed unchanged; the server decrypts the signature to reconstruct the original thinking for prompt construction.

→ A cancelled parallel batch modifies the assistant message's thinking sequence mid-flight; the next request fails the unmodified-sequence requirement → 400, and the poisoned message persists → permanent wedge.

## Conclusion

No hook. The defense is behavioral discipline, enforced by reading the rule, not by code:

- `tool-use.md` Rule 6: one Bash tool_use block per response, all Bash-class actions chained inside it; a Bash block always travels alone; Read/Write/Edit/Grep/Glob may sequence in one response (e.g. Read→Edit) since only Bash carries the cancel cascade.
- Do NOT re-attempt a hook for this problem. The detection point does not exist.
