## Task A — Git index.lock advice block (tool_result)

### What we stripped

The exact 5-line block hardcoded in git's `lockfile.c`:
```
Another git process seems to be running in this repository, e.g.
an editor opened by 'git commit'. Please make sure all processes
are terminated then try again. If it still fails, a git process
may have crashed in this repository earlier:
remove the file manually to continue.
```

This block appears in Bash `tool_result` content when the beads auto-export hook calls `git add` while another git process holds `index.lock`. The variable line that precedes it:
```
Warning: auto-export: git add failed: exit status 128: fatal: Unable to create '<repo-path>/.git/index.lock': File exists.
```
is NOT stripped — it contains the variable repo path and is the actionable signal.

### Why the block is constant

git `lockfile.c` hardcodes this text. It contains no variable data (no path, no PID, no timestamp). It is identical across all repos, all git versions that support index.lock, all OS. Exact multi-line literal match (`str.replace`, no regex) is the right approach — correct, zero false-positives, no pattern drift risk.

### Design decisions

- **New module `strip_git_lock.py`** — mirrors `strip_hook_prefix.py` exactly (fast-path marker, 4 content-shape traversal, returns `(new_content, removed_chunks)`). Keeps each strip concern in its own module.
- **Fast-path marker** `'Another git process seems to be running'` — cheap `in` check before doing any replacement work.
- **Trailing `\n` handling** — tries `_GIT_LOCK_ADVICE + '\n'` first (standard git output ends with newline), then `_GIT_LOCK_ADVICE` alone (edge case). This leaves the warning line with exactly one trailing `\n` and no double blank line after the strip.
- **Guard** `_content_contains` (not `_top_level_content_contains`) — the block appears inside `tool_result` content, so we must descend.
- **Pass position** after `_apply_hook_prefix_strip` in `rules.py` orchestrator — both strip `tool_result` noise, no ordering dependency.
- **mod-name** `stripped_git_lock_advice`; **strip_vocab.py** rule `'GL'` with marker `'Another git process seems to be running'`.

## Task B — Env-context system-reminder (userEmail / currentDate)

### What we stripped

CC injects this SR block on nearly every request:
```
<system-reminder>
As you answer the user's questions, you can use the following context:
# userEmail
The user's email address is brunowinter7934@gmail.com.
# currentDate
Today's date is 2026-05-30.

      IMPORTANT: this context may or may not be relevant to your tasks. You should not respond to this context unless it is highly relevant to your task.
</system-reminder>
```
334 chars of inner text per request, never useful to the proxy model.

### The `_PRESERVE_PREAMBLE` ordering problem

`strip_sr.py::_apply_sr_strip._replace` has a guard:
```python
if inner.startswith(_PRESERVE_PREAMBLE):
    return full  # preserve CLAUDE.md context block
```
where `_PRESERVE_PREAMBLE = "As you answer the user's questions, you can use the following context:"`.

The env-context block starts with EXACTLY this preamble. Without the pre-guard check, the guard fires first and returns `full` (preserve) — the env-context block is never stripped. The fix: check the env-context pattern BEFORE the preamble guard, so the specific full-block match wins.

### Why dedicated regex check, not a template entry

The `_SR_TEMPLATES` dispatch runs AFTER the `_PRESERVE_PREAMBLE` guard (line order: guard → `_match_template`). A template entry for env-context would never be reached — the guard fires first. Even restructuring to put template dispatch before the guard would not work: templates use `startswith` on identifiers, which cannot express "full block with date regex". A `re.fullmatch` against the entire stripped `inner` text is the only mechanism that:
1. Fires before the guard (by being placed before it)
2. Distinguishes env-context (specific email + IMPORTANT footer) from CLAUDE.md context (same preamble, completely different body)

### Regex

```python
_ENV_CONTEXT_RE = re.compile(
    r"As you answer the user's questions, you can use the following context:\n"
    r"# userEmail\n"
    r"The user's email address is brunowinter7934@gmail\.com\.\n"
    r"# currentDate\n"
    r"Today's date is \d{4}-\d{2}-\d{2}\.\s+"
    r"IMPORTANT: this context may or may not be relevant to your tasks\. "
    r"You should not respond to this context unless it is highly relevant to your task\.",
)
```

- Email `brunowinter7934@gmail.com` matches literally (the only email this proxy ever sees).
- Date uses `\d{4}-\d{2}-\d{2}` (the only varying field).
- Whitespace between date sentence and IMPORTANT line uses `\s+` (tolerates minor indentation changes in future CC updates — the only brüchige Stelle).
- All other text is literal.
- `re.fullmatch(inner)` — requires the ENTIRE stripped inner text to match; no partial match possible.

### No new strip_vocab.py entry needed

The env-context check fires inside `_apply_sr_strip`, which is called by the existing `_apply_final_sr_pass`. No new mod-name is added to `modifications[]`. The stripped block is captured in `stripped_msg_removed` as part of the existing ALL-class final-pass tracking. Attribution is `None` from `attribute_chunk` — same as any other SR caught by the catch-all final pass. This is correct; the final pass is the catch-all by design.

## Task C — bd informational auto-import/export lines (tool_result)

### Investigation approach

Grepped two live proxy logs (`api_requests_opus_monitor_cc_1780173436.jsonl` + `api_requests_worker_f93afc17_proxy-strip_1780175908.jsonl`) and ran `strings /usr/local/bin/bd` to extract all bd format-string messages. The log grep confirmed the 4 user-listed variants; the strings extraction revealed 8 additional variants.

### Complete variant set

**From live logs (confirmed in tool_result):**
- `- auto-importing N bytes from /P/.beads/issues.jsonl into empty database` (with `- ` prefix)
- `- auto-imported N issues from /P/.beads/issues.jsonl`
- `- auto-imported N issues into empty database`
- `Exported N issues to /P/.beads/issues.jsonl` (no `- ` prefix)

**Additional from bd binary strings:**
- `auto-imported N issues and N memories from /P/.beads/issues.jsonl`
- `Exported N issues and N memories to /P/.beads/issues.jsonl`
- `auto-export: wrote N issues [and N memories] to /P/.beads/issues.jsonl`
- `auto-export: no changes since last export` (exact literal)
- `auto-export: throttled (last export 5m ago, interval 15m)`
- `auto-export: skipping [reason]`
- `auto-import: N issues [, N memories] from /P/.beads/issues.jsonl (upgrade recovery, GH#2994)`

**NOT stripped (error/warning conditions):**
- `Warning: auto-export: git add failed: …` (Task A strip; warning line preserved)
- `Warning: auto-export failed: …`
- `Warning: auto-export skipped: failed to get current commit: …`
- `warning: auto-import: failed to parse %s: %v`

### Design decisions

- **`- ` prefix handling:** Import variants (`auto-importing`, `auto-imported`) appear with `- ` bullet prefix in hook-captured tool_result; export variants (`Exported`, `auto-export:`) appear without. `^(?:- )?` in the regex handles both. Evidence: live log normalized keys.

- **Three fast-path markers required:** `'issues.jsonl'` covers all path variants. `'auto-export:'` covers no-path status lines (no changes/throttled/skipping/wrote). Neither covers `auto-imported N issues into empty database` (no path, no `auto-export:`). Added `'into empty database'` as third marker. Rules.py guard uses `any(_content_contains(…, m) for m in _BD_NOISE_MARKERS)` — imports the constant rather than hardcoding in rules.py.

- **Error discrimination:** regex alternation starts at `auto-import`/`auto-export`/`Exported` — never at `Warning:` or `warning:`. The only way to accidentally match a warning line would be if it started with one of those exact strings, which none of them do (they all start with `Warning:` or `warning:`).

- **`into empty database` third marker:** Specific enough to bd output that false-positive risk is negligible. No known CC/git/bash output contains this phrase. The marker was confirmed necessary by the failing test case before the fix.

- **Single `_BD_NOISE_RE` with 9-branch alternation:** All variants in one compiled regex — single `re.sub` call per text block. `\n?` at end consumes trailing newline (no orphan blank line). `(?: and \d+ memories)?` covers both with/without memories in one branch.
