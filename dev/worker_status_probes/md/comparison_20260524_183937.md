# Worker Status Sensor Comparison — 2026-05-24

## 1. Methodology

**Problem:** `_worker_detect_status()` in `tmux_spawn.sh` (and `discover.py`) demotes
`hooks.json='working'` to `idle` when `(now - jsonl_mtime) > 10s`. CC only writes JSONL
on message completion; long thinking phases (Caramelizing/Concocting) keep JSONL stale
indefinitely, triggering false-idle demotes minutes into a real turn.

**Goal:** Identify a pane-level sensor immune to the JSONL gap, robust to tmux version
changes, and cheap to query on every `_tick`.

### Target sessions

| Session | Role | Expected state |
|---|---|---|
| `worker-Monitor_CC-ccwrap-phase1` | Worker: completed Phase B | **idle** |
| `worker-searxng-filter-cli` | Worker: hit context limit | **idle** |
| `monitor_cc_f93afc17` | Opus main session | **idle-waiting** (waiting for worker response) |
| `worker-Monitor_CC-status-probe` | This worker: actively executing bash | **working** (supplemental run) |

### Probe runs

**Main run** (`20260524_183937`): 120 seconds, all 3 worker/opus targets. Duration 18:39:37–18:41:37.
During this window all three sessions were idle (Opus waiting for Phase B to complete).
Captured clean idle baseline across all three sensors.

**Supplemental run** (`supp`): 60 seconds, `status-probe` (working) + Opus.
`status-probe` was actively executing bash tool calls throughout — continuous pane writes.
Captured clean working baseline across all three sensors.

## 4. Per-sensor verdict

### Probe A — `window_activity` (tmux format variable)

**Detection accuracy:**

| State | active_sec / total | Rate |
|---|---|---|
| Working (status-probe, 60s) | 59/60 | 98.3% |
| Idle — ccwrap | 1/120 (t=0 artifact) | 0.8% effective |
| Idle — filter | 1/120 (t=0 artifact) | 0.8% effective |
| Idle — opus-main | 0/120 | 0.0% |

Idle sessions: `window_activity` is a Unix timestamp that advances only when bytes are
written to the window. Static panes (zsh prompt, CC waiting) never increment it; the
delta is 0 for every sample. Working sessions: delta=1 on 59/60 samples (98.3%); the
one miss is a startup artifact (init + first poll within the same second boundary).

**Refutes the old claim:** The comment in `tmux_spawn.sh:108` says "Previously used
window_activity, which was bumped by CC UI updates (spinner, cursor blinks) → unreliable."
This probe disproves the cursor-blink claim: a CC session showing the idle prompt and
spinner (opus-main, ~10min stale by probe start) recorded delta=0 for all 120 samples.
The old code may have encountered a different issue (possibly wrong comparison logic or
non-zero initial delta treated as working). Empirically, idle panes produce zero `window_activity`
delta and working panes produce near-100% delta.

**Cost to query:** One subprocess call per session per tick:
```bash
tmux display-message -t <session>:^ -p '#{window_activity}'
```
~2–5ms per call on macOS. No setup, no teardown, no helper processes.

**Robustness:** Depends on tmux format variable name `window_activity` (stable since tmux 1.8,
tmux 3.6a confirmed). Not dependent on CC TUI string content; purely a server-side byte-write
timestamp. Immune to CC UI changes.

**Failure mode:** If CC writes ANY byte to the pane (e.g., a keep-alive or cursor repositioning)
during idle, delta flips to 1. In practice this did not occur across 300 idle-session-seconds
sampled. No false positives observed.

### Probe B — `pipe-pane` + byte-rate file

**Detection accuracy:**

| State | nonzero_sec / total | Avg bytes/sec | Max bytes/sec |
|---|---|---|---|
| Working (status-probe, 60s) | 59/60 | 1,289 B/s | 18,714 B/s |
| Idle — all sessions (120s each) | 0/120 | 0 | 0 |

Same 59/60 detection rate as Probe A (t=0 pipe-pane not yet connected). Provides richer
signal: byte volume lets you distinguish "spinner-only idle" from "streaming response" if
a threshold is needed, though the idle baseline is exactly 0 (no threshold needed).

**Cost to query:** Sampling is a `stat()` call on a file (~0.1ms). But setup requires:
1. `tmux pipe-pane -t <session>:0 '<command>'` per session (one-time at worker spawn)
2. `byte_touch.py` running persistently as a subprocess of the tmux server
3. Teardown: `tmux pipe-pane -t <session>:0` (no command) on probe end

**Robustness:** Main failure mode: `byte_touch.py` crashes silently → sensor reports idle
for an active session. Detection of helper death: bytecount file stops growing while pane
is known active. The SIGPIPE on next write will terminate the helper cleanly but the
sampling loop has no death-detection. Requires monitoring the helper's liveness separately.

Additional concern: for multi-pane sessions (e.g., Opus main with 2 panes in window 0),
pipe-pane targets ONE specific pane. If CC activity is in the OTHER pane, bytes are missed.

### Probe C — `tmux -C` control mode

**Detection accuracy:**

| State | nonzero_sec / total | Avg events/sec | Avg bytes/sec (escaped) |
|---|---|---|---|
| Working (status-probe, 60s) | 60/60 | 13.5 ev/s | 2,128 B/s (escaped) |
| Idle — all sessions (120s each) | 0/120 | 0 | 0 |

100% detection rate — catches the first second because the control mode handshake and
any buffered `%output` events arrive in the initial burst (169 events at t=0). Byte
counts in Probe C are escaped-payload lengths (tmux octal-escapes binary bytes), running
~1.65× larger than actual byte counts (probe B comparison: 77K actual vs 127K escaped).
Use event count as the primary signal, not bytes.

**Cost to query:** One persistent subprocess per session + one reader thread per session:
```python
proc = subprocess.Popen(['tmux', '-C', 'attach-session', '-t', session],
                        stdin=PIPE, stdout=PIPE, stderr=DEVNULL)
```
Confirmed working without a PTY. Each `%output` event is one line from `proc.stdout`.
CPU: low (blocking readline); memory: one thread + pipe buffer per session.

**Robustness:** Geppetto docs note "window switching becomes slow with multiple control
mode clients" (1–3s lag). In our setup each session gets ONE client (not multiple clients
per session), so the cross-session contention noted in their multi-runner architecture
does not apply. However: tmux `-C` vs `-CC` semantics differ between tmux versions;
3.6a confirmed. The `%extended-output` variant (newer tmux) is handled by the parser.
If the subprocess dies (network, tmux restart), the reader thread exits silently. Requires
restart logic for production use.

**Multi-pane sessions:** Control mode receives events from ALL panes in the session. For
Opus (5 windows), filtering to window 0 pane IDs (as implemented) isolates CC activity
from bead-tracker noise in windows 3–4.

## 5. Recommendation

**Probe A (window_activity) wins.** Implementation: replace the JSONL-mtime demote rule
with a `window_activity` staleness check.

### Why A beats B and C

| Criterion | A | B | C |
|---|---|---|---|
| Detection (working) | 98% | 98% | 100% |
| Detection (idle) | 100% | 100% | 100% |
| Setup overhead | none | pipe-pane + helper process per session | subprocess + thread per session |
| Query cost | ~2ms (subprocess) | ~0.1ms (stat) | ~0ms (in-memory counter) |
| Failure modes | near-none | helper crash → false idle | subprocess crash → silent gap |
| Impl. footprint (tmux_spawn.sh) | 2 lines | 20+ lines + helper script path | 50+ lines |
| Impl. footprint (discover.py) | 1 subprocess call in tick | stat() call + init path | background thread pool |
| tmux version sensitivity | stable since 1.8 | stable | -C semantics version-sensitive |

Probe C has 100% detection but costs the most: persistent subprocess + thread per session,
restart logic for failures, and multi-pane filtering complexity. The 2% improvement over A
(one missed t=0 startup artifact) does not justify this. Probe B sits between: lower query
cost than A after setup, but helper process liveness is a silent failure risk.

### Proposed implementation change

**In `tmux_spawn.sh:_worker_detect_status()` (lines 180–194):**

Replace the current JSONL-age demote:
```bash
if [ "$hook_status" = "working" ]; then
    now=$(date +%s)
    jsonl_mtime=$(stat -f "%m" "$jsonl")
    jsonl_age=$((now - jsonl_mtime))
    if [ "$jsonl_age" -gt "$idle_threshold" ]; then
        echo "idle"
    else
        echo "working"
    fi
fi
```

With a `window_activity` staleness check:
```bash
if [ "$hook_status" = "working" ]; then
    wa=$(tmux display-message -t "${session}:^" -p "#{window_activity}" 2>/dev/null || echo "0")
    now=$(date +%s)
    wa_age=$((now - wa))
    if [ "$wa_age" -gt "$idle_threshold" ]; then
        echo "idle"
    else
        echo "working"
    fi
fi
```

**In `discover.py` (line 147):**

Replace `(now - mtime) > WORKING_THRESHOLD_SECS` with:
```python
wa = _get_window_activity(tmux_session)  # tmux display-message format
wa_age = now - wa
if status == 'working' and wa_age > WORKING_THRESHOLD_SECS:
    status = 'idle'
```

`idle_threshold` / `WORKING_THRESHOLD_SECS` stays at 10s. The semantic changes:
- Old: "CC silent in JSONL for >10s → demote" (blind to thinking phases)
- New: "CC pane silent for >10s → demote" (Caramelizing writes every ~1s → never demotes; crash/zsh goes silent immediately → demotes correctly)

### Why the "cursor blinks bump window_activity" concern is resolved

The old implementation rejected window_activity based on a belief that CC's idle spinner
or cursor blinks would keep `window_activity` fresh even for idle sessions. This probe
disproves it: the idle CC session (Opus main, showing spinner, static prompt) recorded
delta=0 for all 120 samples. A crashed session (zsh prompt) would be identical. The fix
is safe to implement.

### Known limitations of this probe

1. The "working" baseline is `status-probe` (bash tool calls), not CC's thinking phase.
   Direct measurement of Caramelizing/Concocting behavior was not captured (both workers
   were idle by probe start). The Phase A empirical check (window_activity age=0s for a
   Caramelizing session) provides supporting evidence.
2. Opus main session showed zero activity during both probe windows — confirmed waiting,
   not working. No working Opus data collected in either run.
3. Supplemental probes used `start_new_session=True` (detached Python subprocesses);
   actual termination timing depends on OS scheduling, not wall-clock measurement.
